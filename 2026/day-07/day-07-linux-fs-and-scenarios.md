# Linux File System Hierarchy (The Most Important Directories)

## Let's understand where things live in Linux

### Core Directories (Must Know)

### `/` (Root)

* The base of the entire Linux file system. Every file and directory starts from here.
* **I would use this when:** I need to navigate the Linux file system from the top-level directory.

### `/home`

* Stores the home directories of regular users. Each user has a personal directory where they can store files without requiring special permissions.
* **I would use this when:** I want to access or manage a user's personal files and directories.

### `/root`

* The home directory of the superuser (root). It is separate from normal users for security reasons.
* **I would use this when:** I need to store or access files that should only be available to the root user.

### `/etc`

* Contains configuration files for the operating system, services, applications, users, and network settings.

**Examples:**

* **`/etc/environment`** – Used to define global environment variables available to all users.

* **`/etc/adduser.conf`** – Used to configure default settings for newly created users, such as the default shell, home directory, and groups.

* **I would use this when:** I need to configure the Linux system or modify application settings.

### `/var/log`

* Stores log files generated by the operating system, services, and applications. These logs are essential for troubleshooting, monitoring, and auditing.

**Examples:**

* **`auth.log`** – Contains authentication and login-related logs.

* **`syslog`** – Contains general system and service logs.

* **I would use this when:** I need to investigate system errors, application failures, or authentication issues.

### `/tmp`

* Stores temporary files created by users, applications, and services. Files in this directory are usually deleted after a reboot.
* **I would use this when:** I need temporary storage for files that are not required permanently.

---

## Additional Directories (Good to Know)

### `/bin`

* A symbolic link to `/usr/bin` on most modern Linux distributions. It contains essential command binaries required for the system.

### `/usr/bin`

* Contains the majority of user command binaries and utilities used in daily Linux administration (e.g., `ls`, `mkdir`, `mv`, `cp`, `cat`).
* **I would use this when:** I want to locate commonly used Linux commands and executables.

### `/opt`

* Reserved for installing optional or third-party software that is not part of the default Linux distribution (e.g., Google Chrome).
* **I would use this when:** I want to check or manage externally installed applications.

# Scenario-Based Practice

## Scenario 1: Service Not Starting

**Problem:** A service called **`ssh`** failed to start after a server reboot.

### Step 1

**Command:**

```bash
systemctl status ssh
```

**Why:** Check whether the service is running, stopped, or failed.

### Step 2

**Command:**

```bash
journalctl -u ssh -n 50
```

**Why:** View the last 50 log entries to identify the reason for the failure.

### Step 3

**Command:**

```bash
systemctl is-enabled ssh
```

**Why:** Verify whether the service is configured to start automatically during boot.

### Step 4

**Command:**

```bash
systemctl list-units --type=service
```

**Why:** Confirm that the service exists and is recognized by the system.

## Scenario 2: High CPU Usage

**Problem:** The application server is slow, and you need to identify the process consuming high CPU.

### Step 1

**Command:**

```bash
top
```

or

```bash
htop
```

**Why:** Display running processes and monitor CPU usage in real time.

### Step 2

**Command:**

```bash
ps aux --sort=-%cpu | head -10
```

**Why:** Display the top 10 CPU-consuming processes and note their Process IDs (PIDs).

### Step 3

**Command:**

```bash
sudo renice +10 -p <PID>
```

**Why:** Lower the process priority so it receives less CPU time without terminating it.

### Step 4

**Command:**

```bash
kill <PID>
```

**Why:** Terminate the process if it is unresponsive or consuming excessive CPU resources.

## Scenario 3: Finding Service Logs

**Problem:** A developer asks where the logs for the **`ssh`** service are located.

### Step 1

**Command:**

```bash
systemctl status ssh
```

**Why:** Check the current status of the service.

### Step 2

**Command:**

```bash
journalctl -u ssh -n 50
```

**Why:** Display the last 50 log entries for the service.

### Step 3

**Command:**

```bash
journalctl -u ssh -f
```

**Why:** Follow the service logs in real time.

---

## Scenario 4: File Permissions Issue

**Problem:** A script is not executing and shows **"Permission denied."**

### Step 1

**Command:**

```bash
ls -l demo.sh
```

**Why:** Check the current file permissions and verify whether execute (`x`) permission is missing.

### Step 2

**Command:**

```bash
chmod +x demo.sh
```

**Why:** Add execute permission to the script.

### Step 3

**Command:**

```bash
ls -l demo.sh
```

**Why:** Confirm that the execute permission has been added successfully.

### Step 4

**Command:**

```bash
./demo.sh
```

**Why:** Execute the script after fixing the permissions.
